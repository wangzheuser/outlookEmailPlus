## 一、需求拆解

Issue #64 包含两个独立子需求：

| 子需求 | 当前状态 | 缺口 |
|--------|----------|------|
| **子需求 A：批量开启邮件通知** | ❌ 完全不存在 | 缺少后端批量 toggle API + 前端批量入口 |
| **子需求 B：批量获取邮件** | ⚠️ 半成品 | 前端已有串行循环（Issue #55），缺少后端批量 API 提供更高效的服务端聚合 |

### 1.1 子需求 A — 批量开启邮件通知

**当前现状**：
- 仅有单账号切换接口：`POST /api/accounts/<id>/telegram-toggle`（`controllers/accounts.py:2661`）
- 该接口操作字段 `telegram_push_enabled`，复用为"通知参与开关"（注释：`notification_dispatch.py:521-523`）
- 前端 `accounts.js:686-714` 只支持逐账号调用 toggle
- 批量操作栏（`index.html:236-245`）中没有通知开关入口

**缺口**：
1. 无后端批量 toggle API（参考现有 `api_batch_update_status` 模式）
2. 无前端批量入口（批量操作栏缺少对应按钮）

### 1.2 子需求 B — 批量获取邮件

**当前现状（Issue #55 已完成部分）**：
- 前端批量拉取入口已存在：批量操作栏 → "批量拉取邮件" 按钮（`index.html:242`）
- 前端批量执行逻辑已实现（`main.js:4442-4538`）：
  - `resolveSelectedAccountsForBatchFetch()` — 解析选中账号
  - `showBatchFetchConfirm()` — 确认弹窗
  - `batchFetchSelectedEmails()` — 串行循环逐账号调用 `GET /api/emails/<email_addr>`
  - `fetchLatestFoldersForAccount()` — 单账号 inbox + junkemail 拉取
- 设计边界：只覆盖标准模式、latest-only、不新增后端 API、不自动切换当前账号

**现有缺口**：
1. **效率问题**：前端串行循环（N 个账号 = N×2 次 HTTP 请求），无并行、无服务端聚合
2. **体验问题**：无真正的进度流（仅 toast 文字更新）、失败信息不结构化
3. **功能缺失**：仅覆盖标准模式，紧凑模式无批量拉取入口

## 二、方案设计

### 方案概览

```
Phase 1 ─── 后端：批量通知开关 API
Phase 2 ─── 前端：批量通知开关入口
Phase 3 ─── 后端：批量获取邮件 API（服务端聚合）
Phase 4 ─── 前端：批量获取邮件升级（接新 API）
Phase 5 ─── 测试与验收
```

### 2.1 方案 A：最小闭环（先做通知批量，邮件批量维持现状）

**范围**：
- 只实现 Phase 1 + Phase 2（批量开启邮件通知）
- 批量获取邮件继续使用 Issue #55 的前端串行方案
- 开发量最小，快速交付通知批量功能

**优点**：
- 改动量约 100 行（后端 ~60 + 前端 ~40）
- 不触及现有批量拉取链路
- 立即可用

**缺点**：
- 批量获取邮件效率问题不解决

### 2.2 方案 B：完整实现（两个都做后端批量 API）

**范围**：
- Phase 1~4 全部实现
- 通知批量：新增 `POST /api/accounts/batch-notification-toggle`
- 邮件批量：新增 `POST /api/emails/batch`（服务端汇聚多个账号的邮件）

**优点**：
- 两个功能都得到真正的后端批量支持
- 效率大幅提升（服务端可并行 + 结构化结果）

**缺点**：
- 邮件批量 API 需要更复杂的设计（并发控制、超时策略、账号类型路由）
- 改动量约 300-400 行

## 七、当前结论

1. **子需求 A（批量开启邮件通知）是纯新增功能**，无历史实现，需要从后端到前端完整搭建。
2. **子需求 B（批量获取邮件）已有前端半成品**（Issue #55），可在不新增后端 API 的情况下直接使用现有前端串行方案，但若追求效率可补充后端批处理 API。
3. **推荐最小闭环**：先做 Phase 1+2（批量通知开关，约 100 行改动），批量获取邮件先维持 Issue #55 现状。
4. **推荐完整方案**：Phase 1~4 全部推进，两个功能都达到真正的后端批量支持。

> ⏭️ 下一步：用户决策方案范围（方案 A 最小闭环 vs 方案 B 完整实现），然后进入开发阶段。
