## 审查清单

### 1. 架构与分层

### 2. 批量通知开关 API 正确性

重点检查：

**参数校验**：
- [ ] `account_ids` 缺失或为空列表 → 400 + 错误码 `ACCOUNT_IDS_REQUIRED`
- [ ] `account_ids` 类型错误（字符串）→ 400 + 错误码 `INVALID_PARAM`
- [ ] `enabled` 缺失 → 默认 `false`，不报错
- [ ] 去重逻辑存在（相同 ID 不重复处理）

**批量执行**：
- [ ] 逐账号调用 `repositories/accounts.py:toggle_telegram_push(aid, enabled)`
- [ ] 存在性校验：不存在的 ID 记录到 `missing_ids`，不报错
- [ ] 部分失败不中断整批
- [ ] 幂等：已开启再开启、已关闭再关闭均不报错

**响应结构**：
- [ ] `success: true`
- [ ] `enabled: bool`（实际设置的值）
- [ ] `updated_count: int`（实际成功数）
- [ ] `failed_count: int`
- [ ] `missing_ids: [int, ...]`（不存在的 ID 列表）
- [ ] `message: str`（中英文）

## 审查原则

1. **不鼓励超范围发挥** — 实现如果刚好做到提示词要求，就是合格的
2. **不因为"还能做更多"而否定符合 TODO 的实现** — 增强项是可选的
3. **重点关注错误路径** — 404/400/缺少参数/无效类型 比正常路径更重要
4. **claimed 保护方向特殊** — 与其他 Issue（如 #60）不同，Issue #64 的 claimed 是"不应保护"的，必须确认没有被误加保护
5. **幂等性是核心安全属性** — 重复调用不能报错、不能产生副作用
6. **与 Issue #55 的边界清晰** — Issue #55 负责前端批量拉取，Issue #64 只负责后端通知批量，两者不应互相渗透
