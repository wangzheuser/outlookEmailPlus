## 3. 测试原则

1. **批量通知开关与池状态无关** — `claimed`/`frozen` 等状态不影响通知开关操作
2. **幂等性** — 重复开启/关闭不报错（复用 `toggle_telegram_push` 幂等逻辑）
3. **部分失败不中断整批** — 某个 ID 不存在只影响该项，不影响其他
4. **与 Issue #55 前端批量拉取不冲突** — 新 API 是增强不是替代

## 4. 关键测试矩阵 — 批量通知开关

### 4.1 API 端点：`POST /api/accounts/batch-notification-toggle`

#### 4.1.1 成功场景

| 编号 | 场景 | 请求体 | 预期 |
|------|------|--------|------|
| N-01 | 批量开启（全部存在） | `{account_ids: [1,2,3], enabled: true}` | 200，`success=true`，`updated_count=3`，DB 中 3 个账号 `telegram_push_enabled=1` |
| N-02 | 批量关闭（全部存在） | `{account_ids: [1,2,3], enabled: false}` | 200，`success=true`，`updated_count=3`，DB 中 3 个账号 `telegram_push_enabled=0` |
| N-03 | 幂等开启（已开启的再开启） | 先开启再发同样请求 | 200，`success=true`，不报错 |
| N-04 | 幂等关闭（已关闭的再关闭） | 先关闭再发同样请求 | 200，`success=true`，不报错 |

#### 4.1.2 部分失败场景

| 编号 | 场景 | 请求体 | 预期 |
|------|------|--------|------|
| N-05 | 部分 ID 不存在 | `{account_ids: [1, 999999], enabled: true}` | 200，`updated_count=1`，`failed_count=1`，`missing_ids=[999999]` |
| N-06 | 混合开启/关闭（已开启的开，已关闭的关） | 部分已开启状态的 ID + 部分已关闭 | 200，统一按 `enabled` 设置，不区分原始状态 |

#### 4.1.3 边界与错误场景

| 编号 | 场景 | 请求体 | 预期 |
|------|------|--------|------|
| N-07 | 空 ID 列表 | `{account_ids: [], enabled: true}` | 400，`success=false`，错误码 `ACCOUNT_IDS_REQUIRED` |
| N-08 | 缺少 account_ids 字段 | `{enabled: true}` | 400，`success=false`，错误码 `ACCOUNT_IDS_REQUIRED` |
| N-09 | ID 类型错误（字符串） | `{account_ids: ["abc"], enabled: true}` | 400，`success=false`，错误码 `INVALID_PARAM` |
| N-10 | 缺少 enabled 字段 | `{account_ids: [1]}` | 200（默认 `enabled=false`），或 400 取决于设计 |
| N-11 | 未登录 | 无 session | 401 或 302 重定向到登录页 |
| N-12 | `claimed` 状态账号 | 含 `pool_status='claimed'` 的账号 | 200，正常操作（通知开关与池状态无关） |
| N-13 | `frozen` 状态账号 | 含 `pool_status='frozen'` 的账号 | 200，正常操作 |

#### 4.1.4 DB 状态验证

| 编号 | 验证点 | 方法 |
|------|--------|------|
| N-14 | `telegram_push_enabled` 字段正确更新 | 查询 DB 确认 |
| N-15 | 开启时 `notification_cursor_states` 写入初始化游标 | 查询 DB 确认 channel=email/telegram 记录存在 |
| N-16 | 开启时 `telegram_last_checked_at` 更新 | 查询 DB 确认字段不为 NULL |
